SEO-2 | Attributs width/height sur les images
[FICHIER] src/index.njk, src/_layouts/base.njk
────────────────────────────────────────────────────────────────────────────────

AVANT (exemples) :
  <img src="/images/logo-violet-final.png" alt="YanaLabs" height="50" />
  <img src="/images/aboutme.jpeg" alt="Yana..." loading="lazy" />
  <img src="/images/Dashboard.webp" alt="Dashboard YanaLabs" loading="lazy" />
  <img src="/images/Icone-Form.png" alt="Formulaire" />
  <img src="/images/Icone-chatgpt.png" alt="ChatGPT" />
  (et tous les outils dans la grille)

APRÈS :
  <img src="/images/logo-violet-final.png" alt="YanaLabs" width="111" height="50" />
  <img src="/images/aboutme.jpeg" alt="Yana..." loading="lazy" width="496" height="528" />
  <img src="/images/Dashboard.webp" alt="Dashboard YanaLabs" loading="lazy" width="712" height="400" />
  <img src="/images/Icone-Form.png" alt="Formulaire" width="36" height="36" />
  <img src="/images/Icone-chatgpt.png" alt="ChatGPT" width="36" height="36" />
  (dimensions réelles de chaque image ajoutées, ratio préservé)

POURQUOI :
  Sans width/height, le navigateur réserve 0px de hauteur puis saute brutalement
  quand l'image arrive — c'est le CLS (Cumulative Layout Shift), l'un des trois
  Core Web Vitals de Google. Un CLS > 0.1 dégrade le classement dans les SERP.
  Les dimensions ajoutées correspondent aux tailles réelles mesurées de chaque image.

SEO-3 | fetchpriority="high" et dimensions sur l'image hero des articles
[FICHIER] src/_layouts/article.njk — ligne 29
────────────────────────────────────────────────────────────────────────────────

AVANT :
  <img src="{{ image }}" alt="{{ title }}" loading="eager" decoding="async" />

APRÈS :
  <img src="{{ image }}" alt="{{ title }}" loading="eager" decoding="async" fetchpriority="high" width="1376" height="768" />

POURQUOI :
  L'image hero est le LCP (Largest Contentful Paint) de la page article.
  "fetchpriority=high" la hisse en priorité maximale dans la file du navigateur,
  gagnant 300-800ms sur le LCP selon la connexion. Les dimensions 1376×768
  correspondent aux images réelles des articles et préviennent le CLS.

SEO-4 | Preconnect et dns-prefetch pour le CDN GSAP
[FICHIER] src/_layouts/base.njk — section <head>
────────────────────────────────────────────────────────────────────────────────

AVANT :
  <link rel="dns-prefetch" href="//fonts.googleapis.com">
  <link rel="dns-prefetch" href="//fonts.gstatic.com">
  <link rel="preconnect" href="https://fonts.googleapis.com">
  <link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>

APRÈS :
  <link rel="dns-prefetch" href="//fonts.googleapis.com">
  <link rel="dns-prefetch" href="//fonts.gstatic.com">
  <link rel="dns-prefetch" href="//cdnjs.cloudflare.com">
  <link rel="preconnect" href="https://fonts.googleapis.com">
  <link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>
  <link rel="preconnect" href="https://cdnjs.cloudflare.com">

POURQUOI :
  GSAP est chargé depuis cdnjs.cloudflare.com. Sans preconnect, le navigateur
  doit résoudre le DNS, ouvrir la connexion TCP et négocier TLS (300-600ms sur
  mobile 4G) au moment où le <script> est atteint. Le preconnect anticipe cette
  connexion pendant le parsing du HTML, éliminant cette latence.

  <script type="application/ld+json">
  {
    "@context": "https://schema.org",
    "@type": "BreadcrumbList",
    "itemListElement": [
      { "@type": "ListItem", "position": 1, "name": "Accueil", "item": "https://yanalabs.fr/" },
      { "@type": "ListItem", "position": 2, "name": "Actualités", "item": "https://yanalabs.fr/actualites/" },

POURQUOI :
  Google affiche le chemin dans les SERP à la place de l'URL brute :
  "yanalabs.fr › Actualités › Google Sheets et automatisation"
  Ce rich result augmente le CTR de 10-30%. Sans ce schema, Google devine
  le chemin de façon approximative.

SEO-7 | Sitemap XML : changefreq et priority + exclusion des pages noindex
[FICHIER] src/sitemap.njk
────────────────────────────────────────────────────────────────────────────────

AVANT :
  {% for page in collections.all %}
    {% if not page.data.excludeFromSitemap %}
    <url>
      <loc>https://yanalabs.fr{{ page.url }}</loc>
      <lastmod>{{ page.date | isoDate }}</lastmod>
    </url>
    {% endif %}
  {% endfor %}

APRÈS :
  {% for page in collections.all %}
    {% if not page.data.excludeFromSitemap and not page.data.noindex %}
    <url>
      <loc>https://yanalabs.fr{{ page.url }}</loc>
      <lastmod>{{ page.date | isoDate }}</lastmod>
      {% if page.url == "/" %}
        <changefreq>monthly</changefreq>
        <priority>1.0</priority>
      {% elif "/articles/" in page.url %}
        <changefreq>never</changefreq>
        <priority>0.8</priority>
      {% elif page.url == "/actualites/" %}
        <changefreq>weekly</changefreq>
        <priority>0.7</priority>
      {% else %}
        <changefreq>monthly</changefreq>
        <priority>0.5</priority>
      {% endif %}
    </url>
    {% endif %}
  {% endfor %}

POURQUOI :
  Les pages noindex (légales) sont automatiquement exclues du sitemap.
  Les priorités guident Googlebot : accueil (1.0), articles (0.8), actualités (0.7).
  "changefreq: never" sur les articles indique qu'ils sont stables — Googlebot
  ne les re-crawle pas inutilement.

GEO-2 | wordCount, articleSection, keywords dans BlogPosting
[FICHIER] src/_layouts/base.njk — schema BlogPosting
[FICHIER] .eleventy.js — nouveau filtre wordCount
────────────────────────────────────────────────────────────────────────────────

AVANT (.eleventy.js) :
  eleventyConfig.addFilter("limit", (array, n) => array.slice(0, n));

APRÈS (.eleventy.js) :
  eleventyConfig.addFilter("limit", (array, n) => array.slice(0, n));
  eleventyConfig.addFilter("wordCount", (content) => {
    if (!content) return 0;
    return content.replace(/<[^>]*>/g, " ").split(/\s+/).filter(Boolean).length;
  });

AVANT (schema BlogPosting) :
  "keywords": "{{ categorie if categorie else 'automatisation no-code, TPE PME' }}",

APRÈS (schema BlogPosting) :
  "articleSection": "{{ categorie if categorie else 'Automatisation No-Code' }}",
  "wordCount": {{ content | wordCount }},
  "keywords": "{{ tags | join(', ') if tags else (categorie if categorie else 'automatisation no-code, TPE PME') }}",

RÉSULTAT dans le HTML généré (article Google Sheets) :
  "wordCount": 1664,
  "articleSection": "Gain Temps",
  "keywords": "articles, automatisation, google sheets, gains temps, no-code",

POURQUOI :
  wordCount (calculé dynamiquement sur le HTML rendu, balises HTML retirées) aide
  les IA à évaluer la profondeur du contenu. articleSection catégorise l'article
  dans le graphe de connaissances. Les keywords sont maintenant extraits des tags
  du front matter plutôt que de la seule catégorie.
