SEO-1 | Meta description de la page d'accueil trop courte et trop vague
────────────────────────────────────────────────────────────────────────────────

[FICHIER] src/index.njk (front matter, ligne 4)

AVANT :
  description: "Gagnez du temps sur vos tâches répétitives grâce à l'automatisation no-code."
  → 79 caractères, aucun mot-clé différenciant, aucune zone géographique

APRÈS :
  description: "YanaLabs automatise vos devis, relances et CRM grâce au no-code (Make, Airtable).
  Audit gratuit et sans engagement pour TPE/PME en Île-de-France. Résultats en 2 semaines."
  → 155 caractères (idéal : 150-160)

POURQUOI :
  Google affiche jusqu'à 160 caractères dans les résultats. La description actuelle

  La version optimisée contient :
    - Les mots-clés cibles : Make, Airtable, no-code, audit gratuit, TPE/PME, Île-de-France
    - Un argument de conversion : "Résultats en 2 semaines"
    - Un bénéfice concret : devis, relances, CRM
  La description influence directement le taux de clic (CTR) dans les résultats Google.
  Un CTR plus élevé améliore indirectement le classement.

SEO-2 | Images sans dimensions width/height → Cumulative Layout Shift (CLS)
────────────────────────────────────────────────────────────────────────────────

[FICHIER] src/index.njk, src/_layouts/article.njk, src/_layouts/base.njk

AVANT :
  <img src="/images/aboutme.jpeg" alt="Yana, fondatrice..." loading="lazy" />
  <img src="/images/Dashboard.webp" alt="Dashboard YanaLabs" loading="lazy" />
  <img src="/images/logo-violet-final.png" alt="YanaLabs" height="50" />

APRÈS :
  <img src="/images/aboutme.jpeg" alt="Yana, fondatrice..." loading="lazy" width="520" height="520" />
  <img src="/images/Dashboard.webp" alt="Dashboard YanaLabs" loading="lazy" width="600" height="450" />
  <img src="/images/logo-violet-final.png" alt="YanaLabs" width="130" height="50" />
  (adapter les dimensions réelles de chaque image)

POURQUOI :
  Sans attributs width/height explicites, le navigateur ne connaît pas la taille
  de l'image avant de la télécharger. Il affiche d'abord 0px de hauteur, puis saute
  brutalement quand l'image arrive. C'est le CLS (Cumulative Layout Shift).
  Le CLS est l'un des trois Core Web Vitals de Google (avec LCP et INP).
  Un CLS > 0.1 dégrade le classement dans les SERP. Google Search Console signale
  ce problème comme "Pages avec une mauvaise expérience".
  Ajouter width + height permet au navigateur de réserver l'espace dès le départ.

SEO-3 | fetchpriority="high" absent sur l'image hero des articles (LCP)
────────────────────────────────────────────────────────────────────────────────

[FICHIER] src/_layouts/article.njk (ligne 29)

AVANT :
  <img src="{{ image }}" alt="{{ title }}" loading="eager" decoding="async" />

APRÈS :
  <img src="{{ image }}" alt="{{ title }}" loading="eager" decoding="async" fetchpriority="high" />

POURQUOI :
  L'image hero est le Largest Contentful Paint (LCP) de la page article — l'élément
  le plus grand visible au chargement. C'est le Core Web Vital le plus impactant.
  "loading=eager" charge l'image sans délai, mais elle reste dans la file normale
  du navigateur. "fetchpriority=high" la hisse en priorité maximale, devant les
  scripts et autres ressources. Sur un article avec image 600Ko+, cela peut gagner
  300-800ms sur le LCP, ce qui fait souvent la différence entre "Good" et "Needs
  Improvement" dans PageSpeed Insights.

SEO-4 | Preconnect manquant pour le CDN GSAP (cdnjs.cloudflare.com)
────────────────────────────────────────────────────────────────────────────────

[FICHIER] src/_layouts/base.njk (section <head>, après les preconnect Google Fonts)

AVANT :
  <link rel="preconnect" href="https://fonts.googleapis.com">
  <link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>
  (GSAP chargé en bas de page depuis cdnjs.cloudflare.com sans preconnect)

APRÈS :
  <link rel="preconnect" href="https://fonts.googleapis.com">
  <link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>
  <link rel="preconnect" href="https://cdnjs.cloudflare.com">

POURQUOI :
  GSAP est chargé depuis cdnjs.cloudflare.com, un domaine externe. Sans preconnect,
  le navigateur doit effectuer en séquence : résolution DNS (50-300ms) + connexion
  TCP (1 aller-retour) + négociation TLS (1-2 allers-retours). Sur une connexion
  mobile 4G, cela peut prendre 300-600ms supplémentaires.
  Le preconnect anticipe cette connexion pendant le parsing du HTML, éliminant
  cette latence avant que le <script> GSAP soit atteint en bas de page.

SEO-5 | Schema BreadcrumbList absent sur les pages article
────────────────────────────────────────────────────────────────────────────────

[FICHIER] src/_layouts/base.njk (dans le bloc {% if layout == "article.njk" %})

AVANT :
  Le fil d'Ariane est affiché visuellement dans article.njk :
    Accueil › Actualités › [Titre de l'article]
  Mais aucun schema JSON-LD ne le décrit pour Google.

APRÈS :
  Ajouter dans le bloc {% if layout == "article.njk" %} en bas de base.njk :

  <script type="application/ld+json">
  {
    "@context": "https://schema.org",
    "@type": "BreadcrumbList",
    "itemListElement": [
      {
        "@type": "ListItem",
        "position": 1,
        "name": "Accueil",
        "item": "https://yanalabs.fr/"
      },
      {
        "@type": "ListItem",
        "position": 2,
        "name": "Actualités",
        "item": "https://yanalabs.fr/actualites/"
      },
      {
        "@type": "ListItem",
        "position": 3,
        "name": "{{ title }}",
        "item": "https://yanalabs.fr{{ page.url }}"
      }
    ]
  }
  </script>

POURQUOI :
  Google lit le BreadcrumbList pour deux raisons :
  1) Il affiche le chemin dans les résultats de recherche à la place de l'URL brute :
     "yanalabs.fr › Actualités › Google Sheets et automatisation"
     Ce rich result augmente le CTR de 10-30% en moyenne.
  2) Il renforce la structure du site dans le Knowledge Graph de Google.
  Sans ce schema, Google peut deviner le fil d'Ariane mais de façon moins fiable.

SEO-7 | Sitemap XML : <changefreq> et <priority> manquants
────────────────────────────────────────────────────────────────────────────────

[FICHIER] src/sitemap.njk

AVANT :
  <url>
    <loc>https://yanalabs.fr{{ page.url }}</loc>
    <lastmod>{{ page.date | isoDate }}</lastmod>
  </url>

APRÈS :
  Stratégie par type de page :
  - Page d'accueil (/) : priority 1.0, changefreq monthly
  - Pages articles (/articles/*) : priority 0.8, changefreq never
  - Page actualités (/actualites/) : priority 0.7, changefreq weekly
  - Pages légales (/legal/*) : priority 0.2, changefreq never

  Implémentation avec un filtre Nunjucks conditionnel sur page.url :
  <url>
    <loc>https://yanalabs.fr{{ page.url }}</loc>
    <lastmod>{{ page.date | isoDate }}</lastmod>
    {% if page.url == "/" %}
      <changefreq>monthly</changefreq>
      <priority>1.0</priority>
    {% elif "/articles/" in page.url %}
      <changefreq>never</changefreq>
      <priority>0.8</priority>
    {% elif page.url == "/actualites/" %}
      <changefreq>weekly</changefreq>
      <priority>0.7</priority>
    {% else %}
      <changefreq>monthly</changefreq>
      <priority>0.5</priority>
    {% endif %}
  </url>

POURQUOI :
  Google déclare officiellement qu'il utilise changefreq et priority comme "hints"
  pour optimiser le budget de crawl. Sur un petit site (<50 pages), l'impact est
  modeste mais permet de signaler clairement que les articles ne changent pas
  (changefreq: never) et que la page d'accueil est la plus importante (priority: 1.0).
  Cela aide Googlebot à ne pas re-crawler des pages stables inutilement.

PRINCIPE FONDAMENTAL DU GEO :

GEO-1 | Schema "speakable" absent — les IA ne savent pas quelle partie citer
────────────────────────────────────────────────────────────────────────────────

[FICHIER] src/_layouts/base.njk (dans le bloc BlogPosting {% if layout == "article.njk" %})

AVANT :
  Le schema BlogPosting ne contient pas de propriété "speakable".

APRÈS :
  Ajouter dans le schema BlogPosting existant :
  "speakable": {
    "@type": "SpeakableSpecification",
    "cssSelector": [
      ".article-description",
      ".article-body blockquote",
      ".article-body h2",
      ".article-toc"
    ]
  }

POURQUOI :
  La propriété "speakable" indique aux systèmes IA (et à Google Assistant) quelles
  zones de la page contiennent les informations les plus importantes et synthétisables.
  Quand Perplexity ou Google AI Overview cherche un extrait à citer pour répondre
  "Comment automatiser Google Sheets ?", il analyse d'abord les zones speakable.
  Sans ce signal, l'IA parse l'ensemble du HTML et peut choisir n'importe quel
  paragraphe, souvent pas le plus représentatif.
  Pointer vers .article-description (intro), les blockquotes et les h2 garantit
  que les résumés IA seront basés sur les messages clés intentionnellement rédigés.

GEO-2 | wordCount et articleSection absents du schema BlogPosting
────────────────────────────────────────────────────────────────────────────────

[FICHIER] src/_layouts/base.njk (schema BlogPosting)

AVANT :
  {
    "@context": "https://schema.org",
    "@type": "BlogPosting",
    "headline": "{{ title }}",
    "description": "{{ description }}",
    ...
    (pas de wordCount ni articleSection)
  }

APRÈS :
  Ajouter dans le schema BlogPosting :
  "wordCount": "{{ content | striptags | wordcount }}",
  "articleSection": "{{ categorie if categorie else 'Automatisation No-Code' }}",
  "keywords": "{{ tags | join(', ') if tags else categorie }}",

  Note : le filtre "striptags | wordcount" nécessite peut-être une configuration
  Eleventy selon la version. Alternative : utiliser un filtre personnalisé dans .eleventy.js.

POURQUOI :
  "wordCount" est une métadonnée lue par les LLMs pour évaluer la profondeur
  du contenu. Un article de 1500 mots sera considéré comme plus autoritaire qu'un
  article de 300 mots sur le même sujet.
  "articleSection" catégorise l'article dans le graphe de connaissances :
  "Gain Temps", "Devis Facturation", etc. Quand un utilisateur demande à ChatGPT
  "articles sur l'automatisation de devis", les pages avec articleSection="Devis
  Facturation" ont un avantage de pertinence structurel.

AVANT :

APRÈS :
  {
    "@context": "https://schema.org",
    "@type": "Organization",
    "name": "YanaLabs",
    "url": "https://yanalabs.fr",
    "logo": "https://yanalabs.fr/images/Favicon.png",

  les schemas Organization et LocalBusiness (actuellement incohérents sur ce point).

GEO-6 | Statistiques sans sources dans les articles — crédibilité réduite
────────────────────────────────────────────────────────────────────────────────

[FICHIER] src/articles/*.md (tous les articles)

AVANT :
  "une erreur de saisie peut coûter 2 à 5% du chiffre d'affaires"
  "la vérification répétée des données prend jusqu'à 3 heures par semaine"
  "Réduction des impayés de 40%"
  (aucune source citée pour aucun de ces chiffres)

APRÈS :
  Option 1 — Source externe crédible :
  "une erreur de saisie peut coûter 2 à 5% du chiffre d'affaires
  <sup><a href="[url-étude-gartner-ou-similaire]" target="_blank" rel="noopener">[1]</a></sup>"

  Option 2 — Source interne transparente :
  "une erreur de saisie peut coûter 2 à 5% du chiffre d'affaires
  (estimation basée sur les audits réalisés par YanaLabs en 2025-2026)"

  Option 3 — Reformuler sans chiffre non sourcé :
  "Dans nos audits, nous observons régulièrement que la vérification manuelle
  des données peut mobiliser plusieurs heures par semaine dans une petite équipe."

POURQUOI :
  Les moteurs IA évaluent la fiabilité des affirmations. Un chiffre précis sans
  source est un signal de contenu non vérifié. Les LLMs ont appris à distinguer
  les claims sourcés des claims non sourcés.
  Conséquences pratiques :
  1) Perplexity affiche des sources et déprécie les affirmations non vérifiables
  2) Google AI Overview préfère les extraits avec des données sourcées
  3) Un utilisateur critique qui vérifie les chiffres et ne trouve pas de source
     perd confiance dans l'ensemble du contenu
  Même une source interne ("selon nos audits") est préférable à l'absence totale.

GEO-7 | FAQ des articles sans schema FAQPage — rich results manqués
────────────────────────────────────────────────────────────────────────────────

[FICHIER] src/_layouts/base.njk (bloc {% if layout == "article.njk" %})

AVANT :
  Le schema FAQPage existe uniquement pour la page d'accueil ({% if faqPage %}).
  Les articles contiennent une section ## FAQ avec des questions/réponses
  mais sans schema JSON-LD.

APRÈS :
  Deux options :

  Option 1 — Front matter structuré (recommandé) :
  Dans le front matter de chaque article :
  faq:
    - q: "Peut-on automatiser Google Sheets sans coder ?"
      r: "Oui, avec des outils comme Make ou Zapier, aucune ligne de code n'est nécessaire."
    - q: "Combien de temps faut-il pour automatiser un tableau ?"
      r: "La mise en place prend généralement 1 à 3 jours selon la complexité."

  Dans base.njk, dans le bloc article, ajouter :
  {% if faq %}
  <script type="application/ld+json">
  {
    "@context": "https://schema.org",
    "@type": "FAQPage",
    "mainEntity": [
      {% for item in faq %}
      {
        "@type": "Question",
        "name": "{{ item.q }}",
        "acceptedAnswer": { "@type": "Answer", "text": "{{ item.r }}" }

  Le schema FAQPage génère des "rich results" dans Google : les questions/réponses

  Avantages mesurés : +20 à +40% de CTR sur les pages avec FAQ rich results.
  Pour les IA : Google AI Overview et Perplexity utilisent activement les schemas
  FAQPage pour générer leurs réponses aux questions directes des utilisateurs.
  Un article sur "automatiser Google Sheets" avec une FAQ structurée a beaucoup
  plus de chances d'être cité par Google AI Overview qu'un article sans.

GEO-9 | Champ "updated" absent des articles — signal de fraîcheur manquant
────────────────────────────────────────────────────────────────────────────────

[FICHIER] src/articles/*.md (front matter de chaque article)

AVANT :
  ---
  date: 2026-04-21
  (pas de champ updated)
  ---
  Le schema BlogPosting génère : "dateModified": "2026-04-21T00:00:00" (= datePublished)

APRÈS :
  ---
  date: 2026-04-21
  updated: 2026-04-25
  ---
  Le schema BlogPosting génère : "dateModified": "2026-04-25T00:00:00"

  Le filtre est déjà en place dans base.njk :
  "dateModified": "{{ updated | htmlDateString if updated else page.date | htmlDateString }}"
  Il suffit d'ajouter le champ updated lors de chaque mise à jour d'article.

POURQUOI :
  Google et les IA valorisent le contenu récemment mis à jour, surtout pour des
  sujets techniques qui évoluent (Make, Airtable, Google Sheets — tous avec des
  mises à jour régulières).
  Un article de 2026 avec dateModified récente sera systématiquement préféré
  à un article similaire avec une date ancienne.
  Pratique recommandée : mettre à jour le champ "updated" à chaque révision de contenu,
  même mineure (correction d'un lien, ajout d'une précision). Cela maintient le signal
  de fraîcheur actif sans avoir à republier l'article.

GEO-10 | Inconsistance du naming "Yana Labs" vs "YanaLabs" — entité fragmentée
────────────────────────────────────────────────────────────────────────────────

[FICHIER] src/index.njk (ligne ~499, section newsletter)

AVANT :
  "J'accepte de recevoir la newsletter et les actualités de Yana Labs."
  (avec espace entre Yana et Labs)

  Autres occurrences à vérifier :
  - Tous les articles .md (vérifier avec : grep -r "Yana Labs" src/)
  - Les pages légales
  - Les schemas JSON-LD (actuellement corrects : "YanaLabs" sans espace)

APRÈS :
  "J'accepte de recevoir la newsletter et les actualités de YanaLabs."
  (sans espace, cohérent avec les schemas et le logo)

POURQUOI :
  La consistance du nom d'entité est fondamentale pour le Knowledge Graph.
  Google et les LLMs construisent leur compréhension d'une entité en agrégeant
  toutes ses mentions sur le web. Si "Yana Labs" (avec espace) et "YanaLabs"
  (sans espace) coexistent, les algorithmes peuvent les traiter comme deux entités
  distinctes, diluant l'autorité et la reconnaissance de chacune.
  Conséquence : si un utilisateur cherche "Yana Labs automatisation", il pourrait
  ne pas trouver les pages optimisées pour "YanaLabs".
  Règle simple : choisir une forme canonique et s'y tenir dans 100% des contenus.
  Recommandation : "YanaLabs" (sans espace), cohérent avec le domaine yanalabs.fr.
